# AI Agent Integration Design

Date: 2026-06-15
Project: 班组排产系统
Status: Draft for user review

## 1. Goal

Add an AI assistant to the existing MES scheduling system. The assistant should answer production questions from local system data and support Feishu/WeChat robot access.

The agreed direction is a two-model design:

- Qwen3.5 chat model for reasoning, tool selection, and final Chinese answers.
- Embedding model for local document and rule retrieval.
- SQLite `data/production.db` remains the source of truth for production facts.

The first version is read-only. It must not modify schedules, approve plans, delete data, trigger MES sync, or execute arbitrary SQL.

## 2. Scope

### In Scope For MVP

- Add a system AI assistant page inside the Flask app.
- Add a unified backend chat API for the AI assistant.
- Connect to OpenAI-compatible Qwen3.5 chat completions API.
- Connect to OpenAI-compatible embeddings API.
- Build a local knowledge index from project documents and operational rules.
- Add read-only production data tools for common questions.
- Enforce existing role and team permissions in every data tool.
- Add AI chat logs for traceability.
- Add Feishu and WeChat robot webhook entry points after the web assistant is stable.
- Store model, bot, and feature settings in `system_settings`.

### Out Of Scope For MVP

- Automatic schedule edits.
- Automatic plan approval or rejection.
- Automatic MES sync execution.
- Arbitrary SQL generated by the model.
- Large external vector database deployment.
- Multi-step autonomous workflows.
- Fine-tuning models.

## 3. Recommended Architecture

The AI feature should be isolated from the current large `app.py` by adding focused utility modules. `app.py` only exposes routes and delegates AI work to the new modules.

```text
Web AI Assistant / Feishu / WeChat
        ↓
Flask route layer
        ↓
Permission context builder
        ↓
AI agent service
        ↓
Knowledge retrieval using embeddings
        ↓
Read-only production data tools
        ↓
SQLite production.db
        ↓
Qwen3.5 final answer
```

## 4. Proposed Files

### New Backend Modules

- `utils/ai_client.py`
  OpenAI-compatible client for `/v1/chat/completions` and `/v1/embeddings`.

- `utils/ai_agent.py`
  Main orchestration: receives a question, retrieves knowledge, chooses tools, calls Qwen3.5, and formats the answer.

- `utils/ai_tools.py`
  Read-only business tools. All database access is parameterized and permission-filtered.

- `utils/ai_knowledge.py`
  Document discovery, chunking, embedding generation, similarity search, and index rebuild.

- `utils/ai_bots.py`
  Feishu/WeChat signature verification, message normalization, and response formatting.

### New Templates

- `templates/ai_assistant.html`
  Internal AI assistant page.

### Existing Files To Touch Later

- `app.py`
  Add routes only. Keep AI logic out of this file.

- `templates/base.html`
  Add a new navigation item with `data-perm="ai_assistant"`.

- `utils/db.py`
  Add table creation for AI-related tables.

- `data/production.db`
  Add new tables and `system_settings` entries through migration or initialization.

## 5. Data Flow

### Internal Web Chat

```text
Logged-in user submits question
  ↓
POST /api/ai/chat
  ↓
Build permission context from session: user_id, role, team_id
  ↓
Retrieve relevant local knowledge chunks by embedding similarity
  ↓
Ask Qwen3.5 to select allowed tools and/or answer from context
  ↓
Run selected read-only tools with enforced permission filters
  ↓
Ask Qwen3.5 to summarize database results and knowledge snippets
  ↓
Save chat log
  ↓
Return answer and optional source references
```

### Robot Chat

```text
Feishu/WeChat webhook receives message
  ↓
Verify bot signature/token
  ↓
Find bound system user or use restricted bot role
  ↓
Call the same AI agent service
  ↓
Return text response to Feishu/WeChat
```

## 6. Model Configuration

Store configuration in `system_settings` so administrators can manage deployment-specific values without code changes.

Recommended keys:

```text
ai_enabled
ai_base_url
ai_api_key
ai_chat_model
ai_embedding_model
ai_vector_store
ai_chroma_mode
ai_chroma_persist_dir
ai_chroma_host
ai_chroma_port
ai_chroma_collection
ai_request_timeout_seconds
ai_max_context_chunks
ai_max_tool_rows
ai_log_retention_days
ai_bot_feishu_enabled
ai_bot_feishu_secret
ai_bot_wechat_enabled
ai_bot_wechat_secret
```

Expected model endpoints:

```text
POST {ai_base_url}/chat/completions
POST {ai_base_url}/embeddings
```

The service is assumed to be OpenAI-compatible. If the chat model and embedding model are served by different base URLs later, add separate `ai_chat_base_url` and `ai_embedding_base_url` keys.

Chroma is the selected vector database for MVP. Use `ai_vector_store=chroma`. For local persistent Chroma, use `ai_chroma_mode=local` and `ai_chroma_persist_dir`. For Chroma server mode, use `ai_chroma_mode=http`, `ai_chroma_host`, and `ai_chroma_port`. Store indexed chunks in a dedicated collection such as `production_ai_knowledge`.

## 7. Knowledge Retrieval Design

The embedding model is used for local documentation and rule retrieval, not for factual production quantities.

### Initial Knowledge Sources

- `AGENTS.md`
- `documentation.md`
- `GANTT_CHART_DESIGN.md`
- MES sync documents
- product definition documents
- image import/download documents
- scheduling, permission, working-hours, and troubleshooting rules

### Chunking Strategy

- Split Markdown and text files by headings first.
- Use approximately 500 to 1,000 Chinese characters per chunk.
- Store source file path, title, chunk text, content hash, and updated time.
- Re-embed only changed chunks.

### Similarity Search

Chroma is the primary vector store. The retrieval flow is:

```text
User question
  ↓
Embedding model creates query vector
  ↓
Chroma searches the `production_ai_knowledge` collection
  ↓
Top matching document chunks are returned with source metadata
  ↓
Qwen3.5 uses those chunks as context
```

SQLite should keep document and chunk metadata for traceability, while Chroma stores vectors, searchable chunk content, and source metadata. If Chroma is temporarily unavailable, the assistant can continue answering database-backed production questions and clearly state that document retrieval is unavailable.

Fallback or future alternatives:

- SQLite `embedding_json` plus Python cosine similarity as an emergency fallback only.
- FAISS for a lightweight local vector index.
- Milvus or pgvector if deployment becomes larger.

## 8. Read-Only Tool Design

The model must not execute free-form SQL. It can only request predefined tools. Each tool validates inputs, uses parameterized SQL, applies role/team filtering, and limits returned rows.

Initial tools:

- `get_work_order_status(order_no=None, product_code=None, date_from=None, date_to=None)`
- `get_schedule_summary(date=None, team_id=None, equipment_id=None)`
- `get_equipment_load(date=None, team_id=None, equipment_code=None)`
- `get_active_alerts(level=None, team_id=None)`
- `get_material_alerts(status=None, team_id=None)`
- `get_work_report_summary(date_from=None, date_to=None, team_id=None)`
- `get_shipping_plan(date_from=None, date_to=None, product_code=None)`
- `get_production_requirements(date_from=None, date_to=None, team_id=None)`
- `get_team_stats(period="daily", date=None, team_id=None)`

Tool result size should be capped by `ai_max_tool_rows`, defaulting to 50.

## 9. Permission Rules

Permission enforcement must happen in the tool layer, not only in prompts.

### Web User Permissions

- `admin`: can query all production data and AI logs.
- `planner`: can query all production data relevant to planning.
- `team`: can query only data for `session.team_id`.

### Robot Permissions

- Unbound robot users get a restricted read-only public summary mode.
- Bound robot users inherit their system account role and team.
- Robot endpoints must verify signatures before processing messages.

### Sensitive Data Restrictions

The AI assistant must not expose:

- Password hashes or credentials.
- API keys and bot secrets.
- Raw `system_settings` values for secrets.
- Unfiltered user account data.
- Arbitrary database schema dumps.

## 10. Database Additions

Recommended tables:

```sql
CREATE TABLE IF NOT EXISTS ai_chat_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    role TEXT,
    team_id INTEGER,
    channel TEXT,
    question TEXT,
    answer TEXT,
    tools_used TEXT,
    knowledge_sources TEXT,
    model TEXT,
    success INTEGER DEFAULT 1,
    error TEXT,
    latency_ms INTEGER,
    created_at TEXT DEFAULT (datetime('now','localtime'))
);

CREATE TABLE IF NOT EXISTS ai_knowledge_docs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_path TEXT NOT NULL UNIQUE,
    title TEXT,
    content_hash TEXT,
    updated_at TEXT DEFAULT (datetime('now','localtime'))
);

CREATE TABLE IF NOT EXISTS ai_knowledge_chunks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    doc_id INTEGER NOT NULL,
    chunk_index INTEGER NOT NULL,
    title TEXT,
    content TEXT NOT NULL,
    chroma_collection TEXT,
    chroma_id TEXT,
    content_hash TEXT,
    updated_at TEXT DEFAULT (datetime('now','localtime')),
    FOREIGN KEY (doc_id) REFERENCES ai_knowledge_docs(id)
);

CREATE TABLE IF NOT EXISTS ai_bot_bindings (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    platform TEXT NOT NULL,
    external_user_id TEXT NOT NULL,
    system_user_id INTEGER NOT NULL,
    is_active INTEGER DEFAULT 1,
    created_at TEXT DEFAULT (datetime('now','localtime')),
    UNIQUE(platform, external_user_id)
);
```

Suggested indexes:

```sql
CREATE INDEX IF NOT EXISTS idx_ai_logs_created ON ai_chat_logs(created_at);
CREATE INDEX IF NOT EXISTS idx_ai_logs_user ON ai_chat_logs(user_id);
CREATE INDEX IF NOT EXISTS idx_ai_chunks_doc ON ai_knowledge_chunks(doc_id);
CREATE INDEX IF NOT EXISTS idx_ai_bot_bindings_platform_user ON ai_bot_bindings(platform, external_user_id);
```

## 11. Routes

Recommended routes:

```text
GET  /ai-assistant
POST /api/ai/chat
GET  /api/ai/logs
POST /api/ai/knowledge/rebuild
GET  /api/ai/settings
POST /api/ai/settings
POST /api/bot/feishu
POST /api/bot/wechat
```

Route decorators:

- `/ai-assistant`: `@login_required`
- `/api/ai/chat`: `@login_required`
- `/api/ai/logs`: `@login_required` plus admin/planner filtering
- `/api/ai/knowledge/rebuild`: `@login_required` and `@admin_required`
- `/api/ai/settings`: `@login_required` and `@admin_required` for writes
- Bot webhooks: no session login, but must perform platform signature verification and binding lookup

Add a new menu item in `templates/base.html`:

```html
<li data-perm="ai_assistant"><a href="/ai-assistant" class="nav-link"><span class="icon">AI</span><span>AI助手</span></a></li>
```

Add `perm_ai_assistant` to `system_settings` with role-specific visibility. Recommended default:

```text

planner: visible

```

This must follow the project rule that new menu permissions are synchronized between `templates/base.html` and `system_settings`.

## 13. Error Handling

The assistant should produce useful errors without exposing internals.

Cases to handle:

- AI disabled.
- Missing model configuration.
- Chat model timeout.
- Embedding model timeout.
- Knowledge index missing or empty.
- Tool validation failure.
- Database locked.
- Robot signature verification failure.
- User not bound to robot account.

Fallback behavior:

- If embeddings fail, continue with database tools and a clear note that document retrieval was unavailable.
- If Qwen3.5 fails, return a friendly failure message and log the error.
- If a tool fails, do not retry destructive actions because all tools are read-only; log and continue if other tools can answer.

## 14. Testing Plan

### Unit Tests Or Scripted Checks

- AI config loading from `system_settings`.
- Chat and embedding client request formatting.
- Knowledge chunking and changed-file detection.
- Cosine similarity ranking.
- Each read-only tool with planner/admin/team permission context.
- Robot signature verification.

### Manual Verification

- Team user can only query own team data.
- Planner/admin can query all teams.
- Model cannot get password/API key/system secret values.
- Model cannot execute arbitrary SQL.
- Common questions return correct database-backed facts.
- Robot answers match web assistant answers for the same bound user.

### Example Acceptance Questions

- `今天有哪些生产预警？`
- `WG7-1 今天排了哪些任务？`
- `订单 X 的完成进度怎么样？`
- `本周哪个班组压力最大？`
- `加班工时怎么算？`
- `MES同步失败 Failed to fetch 怎么处理？`

- Add Feishu webhook.
